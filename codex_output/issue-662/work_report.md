**背景**
- 要件: カスタム単一選択で「オプションクリック時に即クローズし、フォーカスをトリガーへ戻す」こと。
- スコープ: カスタム単一選択のみ（複数選択・ネイティブ `<select>` は対象外）。
- 先行確認: 現実装は既に要件に一致（クリック/Enterで選択→即クローズ、ESC/外側クリックでクローズ、意図的クローズ時はトリガーへフォーカス復帰）。
- 根拠（前段成果物の参照）:
  - オプションクリック: `frontend/src/app/shared/ui/select/ui-select.ts:98`
  - クローズ処理（フォーカス復帰可）: `frontend/src/app/shared/ui/select/ui-select.ts:330`
  - クリック選択→クローズ: `frontend/src/app/shared/ui/select/ui-select.ts:348`
  - 外側クリックでクローズ: `frontend/src/app/shared/ui/select/ui-select.ts:522`
  - Enterで選択→クローズ: `frontend/src/app/shared/ui/select/ui-select.ts:547`
  - ESCでクローズ: `frontend/src/app/shared/ui/select/ui-select.ts:554`

**変更概要**
- コード変更なし（既存挙動が要件を満たすため）。
- キー操作（Enter/ESC）・外側クリックの挙動はそのまま維持。
- 万一、実際の対象が別コンポーネントで未対応なら、クリックハンドラ内で「選択適用 → `closePanel(true)` → トリガーへフォーカス復帰」を最小差分で追加。

**影響**
- 挙動・互換性ともに不変（回帰リスク極小）。
- 複数選択/ネイティブ `<select>` への影響なし（自動クローズしない現行仕様を維持）。
- 依存関係・API 変更なし。アクセシビリティ（フォーカス復帰、役割/状態）も維持。

**検証**
- 単一選択での手動確認:
  - 開く → オプションクリック → 値更新・パネル即クローズ → フォーカスがトリガーへ戻る。
  - 矢印キー移動 + Enter → 選択・クローズ。
  - ESC → クローズ（選択は不変）。外側クリック → クローズ（フォーカスを強制しない）。
  - 無効（disabled）オプションのクリック → 選択/クローズされない。
  - 複数選択/ネイティブ `<select>` → 挙動不変。

**レビュー観点**
- 残存リスク/オープン質問（明示）:
  - Spaceキーの扱い: Enter同等で「選択・クローズ」とするか方針要確認（現状はEnterのみ）。
  - コンポーネント/パスの同一性: このスナップショットでの実ファイル位置が異なる場合、該当パスの共有を要請。
  - 外側クリック判定: 将来ポータル/オーバーレイ化時は包含判定を再検証。
  - まれな blur vs click レース: 事象があればイベント順序の微調整を検討（現状は問題観測なし）。
  - コンテンツ安全性: オプションラベルがリッチHTML化される場合はサニタイズ/エスケープを継続。
  - サーバ安全性: 選択値がサーバ状態に影響する場合、サーバ側の許可リスト/検証とCSRF対策の担保。

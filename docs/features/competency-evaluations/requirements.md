# Competency 評価機能

1. 目的・前提
既存のコンピテンシー評価機能を置き換える形で拡張し、従来の単一選択・単一評価フローはそのまま成立する（完全上位互換）こと。
新機能として「複数コンピテンシーを同時に選択して、まとめて評価できる」こと。
2. ユーザー操作（UI/UX）
2.1 複数選択
ユーザーはコンピテンシー項目を複数選択できること（単一選択も可能）。
選択解除・再選択ができ、選択状態がUI上で明確に分かること。
評価実行（例: 「評価する」ボタン）により、選択中のコンピテンシー一式が評価対象となること。
2.2 評価結果の表示単位
選択した各コンピテンシーごとに、評価結果が「独立したセクション」としてUIに表示されること。
セクション間で内容（スコア/所見/内訳など）が混在しないこと（完全に分離して表示・更新されること）。
複数セクションが並ぶ場合でも視認性・比較しやすさが保たれること（見出し、区切り、折りたたみ等は設計で決定）。
3. API・通信要件
3.1 リクエスト回数
「1回の外部APIリクエスト」で、選択した複数コンピテンシーすべてを評価できること。
追加の評価取得（コンピテンシーごとの個別再リクエスト）は原則行わないこと（再評価など明示操作を除く）。
3.2 リクエスト内容（概念）
APIには「選択したコンピテンシーの一覧（ID/名称等）」をまとめて送れること。
APIレスポンスは「コンピテンシー単位」で結果を返せること（どの結果がどのコンピテンシーに対応するか識別可能であること）。
4. 表示する評価内容（コンピテンシー別・大項目）
各コンピテンシーセクションには、少なくとも以下を表示すること：

総合スコア（コンピテンシー別）
総合所見（コンピテンシー別）
ネクストアクションカード（コンピテンシー別）
評価項目の内訳（コンピテンシー別の詳細）
※「大項目」として、上記4つがコンピテンシーごとに必ず揃って表示されること。

5. 評価項目の内訳（小項目別の詳細表示）
「評価項目の内訳」は、コンピテンシー内の小項目（指定された内訳項目）ごとに分解して表示すること。
各小項目の内訳には、最低限以下を表示すること：
所見
姿勢・意識
行動プラン
スコア
小項目は複数存在し得るため、ユーザーが小項目単位で内容を追えるUI構造にすること（カード/アコーディオン/表などは設計で決定）。
6. データ整合性・上位互換の定義
単一コンピテンシーのみ選択した場合は、従来と同等の情報が表示されること（表示項目の欠落がない）。
複数選択時も、各コンピテンシーの結果は同等の粒度で表示されること（省略・合算しない）。
表示順（選択順/固定順）と更新ルール（再評価時の差し替え/追記）は仕様として明確化し、UIで一貫すること。
